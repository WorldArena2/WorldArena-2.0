# WorldArena 2.0 Track 1 Hugging Face Submission Guidelines

This document is intended for participants. It describes the submission format for Track 1 video-quality evaluation, file organization, the metadata requirements for `model_readme.md`, and how to upload results to a Hugging Face dataset repo for automatic evaluation by the organizers.

The evaluation system will automatically download the single `.tar.gz` or `.tgz` submission package from the public Hugging Face dataset repo you provide, then extract, validate, evaluate, and archive the results. Please prepare your submission strictly according to this document.

## 1. Task Objective

Track 1 evaluates the quality of videos generated by world models. Participants must use the official WorldArena 2.0 test set(https://huggingface.co/datasets/WorldArena/WorldArena2.0/blob/main/dataset_track1.tar.gz) to generate one corresponding `.mp4` video for each episode, then package and submit all videos together with the model description file.

The organizers will always use the following test set for evaluation:

```text
https://huggingface.co/datasets/WorldArena/WorldArena2.0/blob/main/dataset_track1.tar.gz
```

Participants only need to submit the generated videos and model metadata.

---

## 2. Video Generation Requirements

| Item | Requirement |
|------|-------------|
| Video format | `.mp4` |
| Data split | Official WorldArena Track 1 test split |
| Resolution | Recommended: `640x480` or higher |
| Frame rate | Recommended: `24` fps |
| Text-driven length | Recommended to align with the length of the corresponding GT trajectory |
| Action-driven length | Recommended to align with the length of the corresponding GT trajectory |
| File location | All videos must be placed in the `videos/` directory inside the archive |

Use clear, traceable video filenames, for example:

```text
videos/
├── episode_000001.mp4
├── episode_000002.mp4
├── episode_000003.mp4
└── ...
```

> Note: Do not place evaluation videos outside the `videos/` directory.

## 3. Hugging Face Repository Requirements

Participants must create a Hugging Face **dataset** repo and upload one submission archive.

| Item | Requirement |
|------|-------------|
| Repo type | Must be `dataset` |
| Repo access | Must be public and downloadable |
| Authorization | Tokens are not supported; private repos and gated repos are not supported |
| Number of archives | The repo root must contain exactly one `.tar.gz` or `.tgz` archive |
| Archive location | Must be in the repo root, not in a subdirectory |

Valid example:

```text
your-hf-dataset-repo/
└── submission.tar.gz
```

Invalid example:

```text
your-hf-dataset-repo/
├── submission_v1.tar.gz
└── submission_v2.tar.gz
```

```text
your-hf-dataset-repo/
└── release/
    └── submission.tar.gz
```

## 4. Submission Archive Format

The submission package must use the `.tar.gz` or `.tgz` format and contain:

1. `model_readme.md`
2. `videos/`

Recommended structure:

```text
submission.tar.gz
├── model_readme.md
└── videos/
    ├── episode_000001.mp4
    ├── episode_000002.mp4
    └── ...
```

One additional parent directory inside the archive is also allowed, but that parent directory must still contain `model_readme.md` and `videos/`:

```text
submission.tar.gz
└── MyModel_eval/
    ├── model_readme.md
    └── videos/
        ├── episode_000001.mp4
        └── ...
```

To avoid extraction security-check failures, make sure the archive does not contain:

- Absolute paths, such as `/home/user/file.mp4`
- Parent-directory traversal paths, such as `../file.mp4`
- Symbolic links or hard links
- Device files, FIFOs, or other special files
- Multiple `model_readme.md` files

## 5. `model_readme.md` Specification

`model_readme.md` must begin with a YAML metadata block. The automatic evaluation system reads the model name, version, organization, year, open-source status, and control type from this block.

### 5.1 Required Template

```markdown
---
model_name: example_model
version: v1.2
organization: Example Lab
release_year: 2026
source_type: open_source
control_type: text_driven
license: Apache-2.0
paper_url: https://example.com/paper
code_url: https://github.com/example/example_model
---

# example_model v1.2

example_model v1.2 is a text-driven world model for robotic video generation.

## Submission Notes

The submitted videos correspond to the official WorldArena Track 1 test split.
```

### 5.2 Field Requirements

| Field | Required | Valid values or description |
|------|----------|-----------------------------|
| `model_name` | Yes | Model name, such as `example_model` |
| `version` | Yes | Model version, such as `v1.2` or `2026-06-ckpt` |
| `organization` | Yes | Team, school, company, or laboratory name |
| `release_year` | Yes | Four-digit year, such as `2026` |
| `source_type` | Yes | Must be `open_source` or `closed_source` |
| `control_type` | Yes | Must be `text_driven`, `action_driven`, or `hybrid` |
| `license` | No | For example, `Apache-2.0`, `MIT`, or `Proprietary` |
| `paper_url` | No | Paper URL |
| `code_url` | No | Code repository URL |

### 5.3 Field Definitions

`source_type` indicates whether the model is open source:

| Value | Meaning |
|------|---------|
| `open_source` | The model code, weights, or core implementation is publicly available |
| `closed_source` | The model code, weights, or core implementation is not publicly available |

`control_type` indicates how the model is driven when generating videos:

| Value | Meaning |
|------|---------|
| `text_driven` | Primarily driven by text instructions |
| `action_driven` | Primarily driven by action sequences |
| `hybrid` | Uses multiple conditions, such as text and actions, at the same time |

### 5.4 Example File

The organizers provide an example:

```text
https://huggingface.co/datasets/WorldArena/WorldArena2.0/blob/main/examplev1.tar.gz
```

## 6. Packaging and Upload Examples

### 6.1 Prepare the Local Directory

First organize the files into a local directory such as:

```text
example_model_eval/
├── model_readme.md
└── videos/
    ├── episode_000001.mp4
    ├── episode_000002.mp4
    └── ...
```

### 6.2 Create the `.tar.gz` Archive

```bash
tar -czf submission.tar.gz -C example_model_eval .
```

Check the archive contents:

```bash
tar -tzf submission.tar.gz | head
```

You should see entries similar to:

```text
./model_readme.md
./videos/
./videos/episode_000001.mp4
./videos/episode_000002.mp4
```

### 6.3 Upload to a Hugging Face Dataset Repo

Method 1: Upload through the website.

1. Create a dataset repo on Hugging Face.
2. Confirm that the repo is public.
3. Upload `submission.tar.gz` to the repo root.
4. Confirm that the repo root contains only this one `.tar.gz` or `.tgz` submission archive.

Method 2: Upload from the command line.

```bash
huggingface-cli upload <user-or-org>/<dataset-repo> \
  submission.tar.gz \
  submission.tar.gz \
  --repo-type dataset
```

After uploading, submit the Hugging Face dataset repo ID and revision to the organizers, for example:

```text
hf_repo_id: your-org/example-model-track1-submission
hf_revision: main
```

## 7. Pre-submission Checklist

Confirm each item before submitting:

- The Hugging Face repo type is `dataset`.
- The Hugging Face repo is public, does not require a token, and is not gated.
- The repo root contains exactly one `.tar.gz` or `.tgz` file.
- The archive contains `model_readme.md`.
- The archive contains `videos/`.
- `videos/` contains at least one `.mp4` file.
- `model_readme.md` is inside the archive and begins with a YAML metadata block.
- `model_name`, `version`, `organization`, `release_year`, `source_type`, and `control_type` are all filled in.
- `release_year` is a four-digit year.
- `source_type` is `open_source` or `closed_source`.
- `control_type` is `text_driven`, `action_driven`, or `hybrid`.
- The archive contains no absolute paths, `..` paths, link files, or special files.

Quick local-check commands:

```bash
tar -tzf submission.tar.gz | grep -E '(^|/)model_readme\.md$'
tar -tzf submission.tar.gz | grep -E '(^|/)videos/'
tar -tzf submission.tar.gz | grep -E '\.mp4$' | head
```

## 8. Common Invalid Submissions

The following situations will cause automatic evaluation to fail:

| Failure reason | Description |
|----------------|-------------|
| `hf_repo_not_public_or_not_downloadable` | The repo is private, gated, missing, unauthorized, or cannot be downloaded without a token |
| `missing_tar_gz` | The repo root does not contain a `.tar.gz` or `.tgz` file |
| `multiple_tar_gz_found` | The repo root contains multiple `.tar.gz` or `.tgz` files |
| `unsafe_tar_path` | The archive contains absolute paths, `..` paths, links, or special files |
| `extract_failed` | The archive is corrupted or cannot be extracted |
| `missing_model_readme` | `model_readme.md` is missing |
| `metadata_invalid` | The metadata block is missing, fields are missing, or field values are invalid |
| `missing_videos` | The `videos/` directory is missing |
| `empty_video_dir` | The `videos/` directory contains no `.mp4` files |
| `evaluation_failed` | The submission passes validation, but the official evaluation process fails |

---

> Reminder: Each model name may be submitted at most three times. Submissions beyond the third will be considered invalid. Repeated submissions during peak periods may lower evaluation priority, so please complete the local checks before submitting.
