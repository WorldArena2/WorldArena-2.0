# WorldArena 2.0 Track 2：世界模型RL环境赛道

**[中文文档](https://github.com/WorldArena2/WorldArena-2.0/blob/main/assets/track2_benchmark_description_zh.md)**

**[English Document](https://github.com/WorldArena2/WorldArena-2.0/blob/main/assets/track2_benchmark_description_en.md)**

## 1. 赛道目标

Track 2 评估动作条件世界模型（World Model，WM）作为机器人
Model-Based Reinforcement Learning（MBRL）环境的实际价值。

参赛选手提交的不是机器人策略或奖励模型，而是一个可被官方评测端调用的世界
模型推理服务。给定视觉历史、可选状态/语言条件和未来动作序列，服务应生成与
动作逐步对应的未来视觉序列。官方使用这些生成结果进行策略 rollout、奖励估计
和策略优化，再在独立保留环境中评估优化后的策略。

本赛道关注的核心问题是：

> 使用该世界模型产生的交互数据，能否在固定训练预算下稳定提升机器人策略，
> 且这种提升能否迁移到独立评测环境？

视频观感不是最终目标。时序一致性、动作可控性和任务相关动力学，应最终体现为
对策略学习的有效支持。

## 2. 评测对象与职责

### 2.1 参赛选手

- 托管或打包符合公开 API 的世界模型服务；
- 根据请求中的历史观测和动作条件生成未来 RGB 帧；
- 在约定的资源、延迟和可用性预算内稳定响应；
- 保证相同模型版本和随机种子具有可复现行为；
- 声明服务能力、模型版本及必要的第三方依赖。

### 2.2 官方评测端

- 提供初始条件、动作候选及必要的任务条件；
- 统一动作适配、策略 rollout、奖励计算和 MBRL 更新；
- 固定各参赛提交的策略起点、优化算法和交互预算；
- 在与参赛服务隔离的保留环境中进行下游策略评估；
- 记录协议正确性、有效样本率、运行性能和评测结果。

### 2.3 接口边界

参赛服务只负责预测未来视觉观测，不应返回或替代官方组件负责的动作、奖励、
成功判定、终止信号或最终分数，也不需要访问官方策略参数、奖励模型、保留评测
环境或未公开数据。响应中的自定义诊断字段不会参与策略训练或排名。

## 3. 评测流程

```text
官方初始条件
  -> 固定起点的机器人策略产生动作候选
  -> 参赛世界模型服务生成未来视频
  -> 官方奖励与 MBRL 流程更新策略
  -> 独立保留环境评估更新后的策略
  -> 汇总策略效用、稳健性和服务有效性
```

评测采用闭环、多轮动作块 rollout。后续请求可以使用前一轮生成结果作为新的
视觉历史，因此模型需要控制长时滚动中的漂移，而不只是生成单段视频。

为保证公平性，官方对所有提交固定：

- [初始策略 checkpoint](https://huggingface.co/WorldArena/WorldArena2.0/tree/main/pi05_adjust_bottle)；
- [RL奖励函数 checkpoint](https://huggingface.co/WorldArena/WorldArena2.0/tree/main/reward_model)；
- 任务：adjust bottle
- [WM微调数据集](https://huggingface.co/datasets/TianxingChen/RoboTwin2.0/blob/main/dataset/adjust_bottle/aloha-agilex_clean_50.zip)

训练期间的动作由当前策略动态产生，不预先固定整条轨迹。不同世界模型会产生
不同未来观测，策略更新后也可能产生不同后续动作；这种闭环差异正是本赛道需要
评价的能力。

## 4. 输入与输出

每个预测样本的规范输入包括：

- 按时间从旧到新排列的 RGB 历史帧；
- 与历史帧转移对应的动作历史；
- 需要预测的未来动作序列；
- 可选的机器人状态历史；
- 可选的自然语言任务描述；
- 官方指定的随机种子和不透明请求标识。

规范输出包括：

- 与未来动作逐步对齐的 RGB 预测帧；
- 服务版本和请求关联信息；
- 可选、非评分用途的运行诊断信息。

图像、动作、状态的编码、shape、对齐规则和空值规则以
[API 文档][world_model_service_api_zh.md](https://github.com/WorldArena2/WorldArena-2.0/blob/main/assets/track2_world_model_service_api_zh.md)为准。官方将在评测前提供需要支持的
profile；公开示例仅用于接口联调，不代表保留评测分布。

## 5. 评价内容

### 5.1 下游策略效用

核心评价内容是使用参赛世界模型进行 MBRL 后，策略在保留环境中的表现，以及
相对优化前策略的提升。最终评价以闭环策略收益为主，不以单独的逐帧重建指标或
视频观感替代。

### 5.2 动作条件与长时稳健性

官方将检查不同动作序列是否产生合理、可区分且时序连续的结果，并评估多轮闭环
生成中的误差累积、状态退化和异常敏感性。

### 5.3 泛化与一致性

评测覆盖保留的初始条件、任务条件和随机种子。模型不应依赖样本顺序、请求标识
或固定公开样例推断预测内容。

### 5.4 服务有效性

协议通过率、有效响应率、延迟、超时、可复现性和资源合规性用于资格检查、结果
有效性判断或同分处理。通过接口检查本身不代表世界模型质量达标。

正式评测使用冻结且版本化的规程。保留任务、初始状态、动作序列、随机种子、
奖励实现、样本数量及聚合参数不在公开联调数据中提供。

## 6. 公平性与可复现性

- 所有参赛服务使用相同版本的官方评测器和等价的策略优化预算；
- 每次正式评测绑定一个不可变的 `model_version`；
- 评测器传入的 `seed` 必须用于对应样本的随机生成；
- 同一 `request_id` 的重试必须幂等，不得被重复计为新样本；
- 正式评测期间不得静默更新权重、预处理、后处理或依赖版本；
- 官方可以进行重复运行、异常样本复核和输出一致性检查；
- 服务端时间、请求次序、来源地址和不透明标识不得作为预测内容的条件。

## 7. 数据与评测完整性

参赛选手必须遵守以下要求：

- 不保存、传播或将官方请求中的非公开数据用于后续训练；
- 不通过人工干预、远程人工选择或评测期间在线标注生成结果；
- 不探测、反推或利用官方策略、奖励模型、保留任务及评分实现；
- 不根据公开联调样例硬编码输出或识别正式评测请求；
- 不在响应中嵌入可执行内容、外部资源引用或与预测无关的数据；
- 仅使用已申报且具备合法授权的数据、模型和依赖。

除完成当前请求所必需的短期内存或缓存外，服务应默认不持久化请求内容。正式
评测结束后，应按赛事数据政策删除临时缓存和日志中的请求载荷。

## 8. 服务与提交要求

参赛选手可提交官方可访问的 HTTPS 服务，或按赛事通知提交由官方托管的容器
镜像；两种方式必须暴露同一 API。每个提交至少包含：

- 服务基础 URL 或容器镜像的不可变摘要；
- API 认证信息，通过指定安全渠道单独提交；
- `model_version`、`api_version` 和能力清单；
- 启动、健康检查和必要资源说明；
- 模型、数据和第三方组件的许可与合规声明；
- 技术联系人和评测窗口内的故障响应方式。

服务必须在整个评测窗口保持版本不变和可访问。基础设施故障可以修复，但任何
影响模型输出的变更均视为新提交，需要重新登记。

## 9. 失败请求与无效样本

以下情况会被记录为失败请求或无效样本：

- 无法连接、超时或非 2xx 响应；
- schema、batch、sample、shape 或 dtype 不合法；
- 输入包含非有限动作值；
- 输出帧数量错误或图像无法解码；
- 请求与响应的 ID 或模型版本不匹配。

评测器可以对可重试故障进行有限次数重试。持续故障、协议违规、版本漂移或数据
完整性违规可导致该次评测中止或提交失效。
